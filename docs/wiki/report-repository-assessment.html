リポジトリ調査レポート

作成日: 2026-03-16
対象: /Users/ryoike/Documents/Graduation Research の現在のワーキングツリー

1. 調査の前提

  • 本レポートは HEAD ではなく、未コミット変更を含む現在のワーキングツリーを対象にした。
  • 主要に確認したもの:
    • README.md
    • runners/, agents/, core/
    • experimental/production_poc/
    • docs/production_poc/
    • scenarios/definitions.yaml
    • tests/
    • results/, observations/
  • ワーキングツリーは dirty であり、agents/judge.py, agents/triage_agent.py, tests/test_judge.py, tests/test_triage_agent.py などの追加・変更が存在する。したがって本評価は「今このディレクトリにある実装」の評価である。

2. 総評

このリポジトリは、現時点では 「LLM を用いた安全な応急復旧の研究評価基盤」 としてかなりよく整理されている。特に、構造化アクション、事前検証、事後検証、ロールバック、シナリオ化された障害注入、結果保存までが一連の実験フローとして成立している点は強い。

一方で、卒業研究案である 「LLMマルチエージェント協調を用いたDCインフラ応急修復システム」 という題目に対しては、現状はまだ 「安全な復旧実験基盤 + 低リスク本番PoC」 の段階であり、本格的な DC インフラ応急修復システムそのもの と言うには不足がある。特に不足しているのは、DCインフラらしさ、本格的なマルチエージェント協調、本番環境での実修復能力の広さ である。

3. リポジトリの現状整理

3.1 主系: Docker Compose 上の安全な応急復旧実験基盤

主系は agent.py / runners/run_single.py から起動する単一エージェント型の評価基盤である。全体フローは以下の通り。

  1. sensor が docker compose ps、HTTP チェック、ログ、設定ファイル断片を収集する
  2. triage が open-world の故障ドメイン仮説と修復スコープを生成する
  3. 必要時のみ追加観測を 1 回実施する
  4. worker が構造化アクションの修復計画を返す
  5. verifier がスコープ逸脱や危険操作を事前検証する
  6. executor が許可されたアクションのみ実行する
  7. postcheck と必要時の rollback を実施する

この系は自由な shell 実行ではなく、edit_file, restart_compose_service, rebuild_compose_service, run_config_test, run_health_check などの 構造化アクションのみ を許可している。安全性を研究テーマの中心に据えた設計である。

3.2 マルチエージェント系: minimal loop

multi_agent.py / runners/run_multi_minimal.py では、単一エージェントの安全機構をそのまま使いながら、

  • planner
  • reviewer
  • judge

を直列に回す最小ループが実装されている。流れは概ね

planner -> precheck/execute/postcheck -> reviewer -> judge -> replan

であり、最大ターン数も設定可能である。

ただし、これは 本格的な分業型マルチエージェント というより、逐次的な再計画ループ に近い。専門エージェントの並列協調、役割間の交渉、共有知識ベース、サブタスク分解、複数サービス横断の協調制御はまだない。

3.3 実機寄り PoC: experimental/production_poc/

別系統として、Ubuntu 実ホスト向けの production_poc が存在する。これは

  • startup snapshot (discover)
  • ルールベース監視 (monitor-once)
  • 異常時のみ LLM 解析
  • ActionGuard による行動制限
  • Discord 通知
  • systemd timer による定期実行

を備えた、安全寄りの PoC である。

この PoC は「本番向け」と言っても、設計思想はかなり保守的であり、デフォルトは propose-only、自動実行できるのは allowlist 済み service restart を最大 1 回 に限られる。つまり、実態としては 監視 + 提案 + ごく低リスクな初動 の段階である。

4. 強み

4.1 安全設計が一貫している

  • 任意 shell を LLM に返させない
  • 編集対象ファイルが限定されている
  • replace_text の 1 箇所一致制約がある
  • precheck で差分量・スコープ・許可アクションを検査する
  • postcheck と rollback がある
  • rollback 後に refresh action を伴う

この一連の設計は、「LLM をそのまま危険な実行主体にしない」という研究上の重要論点にしっかり応えている。

4.2 シナリオ設計が研究向き

scenarios/definitions.yaml には A-U の障害シナリオがあり、単純故障だけでなく、

  • stale log による誤誘導
  • opaque 500
  • visible green / hidden red
  • competing repair choice
  • masked cascade
  • non-commutative な多段障害

まで含まれている。これは卒論で「なぜ multi-agent や verifier が必要か」を議論する材料として非常に良い。

4.3 実験ログ・集計基盤がある

results/ に試行 JSON が保存され、observe_runs.sh と aggregate_observations.py によって観測・集計ができる。研究として再現可能性や比較実験を組み立てやすい。

4.4 本番PoCも完全に机上ではない

production_poc では、実ホストに対して

  • systemd service の状態
  • process / port / HTTP / TCP
  • disk / memory / CPU
  • journal
  • Minecraft 運用形態の差異

まで見ており、実際の運用系へ接続する導線は用意されている。

5. 弱み・不足

5.1 DCインフラ としては対象が狭い

主系は Docker Compose の nginx -> app -> db 3 層構成であり、production PoC 側も Web service, Minecraft, host resource 監視が中心である。したがって、データセンターインフラに典型的な

  • 複数ホスト協調
  • ネットワーク機器や経路制御
  • ストレージ障害
  • クラスタ/フェイルオーバ
  • 電源・冷却・BMC 連携
  • 運用手順や承認フローとの連携

は未対応である。
現状は 「DCインフラ」より「単一ホスト/単一スタックの応急復旧研究基盤」 に近い。

5.2 マルチエージェント協調はまだ浅い

planner, reviewer, judge は存在するが、

  • 並列協調
  • 専門分化した agent 群
  • agent 間の役割交渉
  • shared memory / blackboard 的な世界モデル
  • サービス別担当 agent
  • 根本原因解析 agent と修復 agent の分業

はまだ実装されていない。
したがって「マルチエージェント協調」というより、再計画を伴う直列 multi-turn 制御 と評価するのが妥当である。

5.3 実修復能力は安全優先で狭い

主系でも編集できるのは限られた数ファイルのみであり、production PoC では実質的に 自動実行できるのは allowlist 済み restart_service だけ である。

つまり本番PoCは、

  • config 修正
  • package rollback
  • traffic switching
  • failover
  • service migration
  • multi-host remediation
  • runbook 自動実行

には進んでいない。
このため、題目に含まれる 「応急修復システム」 という言葉に対しては、現状は 「安全な初動対応システム」 の方が近い。

5.4 バックアップ/スナップショットの本実装がない

experimental/production_poc/adapters/backup_provider.py は stub であり、NullBackupProvider が返る。
研究として安全性を強調するには妥当だが、本番応急修復システムとしては大きな弱点である。

5.5 実証範囲に偏りがある

observations/ に残る 47 試行をみると、少なくとも記録済み範囲では:

  • a: 5/5 成功
  • b: 5/5 成功
  • c: 5/5 成功
  • d: 3/5 成功
  • e: 3/3 成功
  • f: 3/3 成功
  • g: 3/3 成功
  • h: 2/3 成功
  • i: 1/1 成功
  • k: 1/1 成功
  • l: 1/1 成功
  • i2: 0/3 成功
  • m: 0/3 成功
  • n: 0/3 成功
  • o: 0/3 成功

となっている。

ここから言えるのは、

  • 単純故障・局所故障には強い
  • 多段障害・マスク障害・再計画前提の難シナリオにはまだ弱い

ということである。
題目が「マルチエージェント協調」を含む以上、本来は i2/m/n/o/r/u のようなケースで優位性を示したいが、現状その部分はまだ発展途上である。

5.6 評価パイプラインに軽いドリフトがある

シナリオ定義は A-U まで拡張されている一方で、評価まわりにはまだ追随しきれていない箇所がある。

  • tests/test_scenarios.py では p/q/r と s/t/u の存在を確認している
  • しかし observe_runs.sh のデフォルト sweep には p/q/r が含まれていない
  • aggregate_observations.py の期待ドメイン対応表は r までで止まっており、s/t/u が未反映
  • README.md でも一部に A-R 前提の説明が残っている

つまり、実装・シナリオ・観測集計・文書が完全同期しているとはまだ言えない。
卒論ではこの種のズレが「評価範囲はどこまでか」という読者の不安に直結するため、早めに揃える価値が高い。

6. テストと検証状況

6.1 テスト数

tests/ には 67 件のテスト関数が存在した。

6.2 実行結果

今回の確認では以下の結果だった。

  • システム Python の pytest -q:
    • python-dotenv 不足により collection error
  • .venv/bin/python -m pytest -q:
    • 65 passed, 2 failed

6.3 失敗内容

失敗した 2 件は tests/test_multi_minimal.py の multi-agent mock シナリオで、アルゴリズム自体というより、reset.sh 実行時に Docker コンテナ名 /target-db が既存コンテナと衝突して復旧環境を作れなかったことによるものであった。

したがって、

  • 単体テスト・PoCテストはかなり整っている
  • ただし Docker 実験環境のクリーンアップ前提には改善余地がある

と評価できる。

6.4 検証強度のまとめ

検証基盤そのものはしっかりしているが、強さには濃淡がある。

  • 単純〜中程度の故障に対しては、テスト・観測ログ・結果 JSON が揃っており、かなり説得力がある
  • hard scenario に対しては、検出精度よりも 最終復旧成功率 がまだ不足している
  • production PoC は unit test と手動検証シナリオが中心で、長期実運用ログや live-run の蓄積はまだ薄い

したがって現状の検証は、「研究プロトタイプとしては十分強いが、完成システムの実証としてはまだ中程度」 とみなすのが妥当である。

7. 卒業研究案との適合性評価

評価対象の題目: 「LLMマルチエージェント協調を用いたDCインフラ応急修復システム」

7.1 要件分解と判定

観点 判定 コメント
LLM を用いた診断・計画 高 single-agent worker、planner、reviewer、judge、triage の役割分離が入り始めている
マルチエージェント協調 中 minimal loop はあるが、本格的な協調・分業・並列化はまだ弱い
応急修復能力 中 実験環境では成立。production PoC では restart 中心で限定的
安全性・検証・ロールバック 高 このリポジトリの最も強い部分
DCインフラ適用性 低〜中 単一ホスト/単一スタック寄りで、DC 特有の複雑性はまだ薄い
実験・再現・評価基盤 高 シナリオ、結果保存、集計、テストがある
実運用PoCとしての完成度 中 監視・通知・低リスク初動はあるが、本格修復系ではない

7.2 総合点

以下の 2 つに分けて評価するのが妥当である。

A. 卒論の「研究基盤」としての完成度

78/100

理由:

  • LLM を安全に使う設計が明確
  • 実験シナリオと評価基盤が充実
  • 単一エージェントから multi-agent minimal への発展線が見える
  • 実機PoCもあり、研究としての具体性が高い

B. 題目どおりの「DCインフラ応急修復システム」としての完成度

58/100

理由:

  • 「LLM」は十分ある
  • 「応急修復」も研究環境では成立している
  • ただし「マルチエージェント協調」はまだ最小構成
  • 「DCインフラ」と呼ぶには対象領域がまだ狭い
  • 実機側は safe monitoring / guarded restart PoC の色が強い

7.3 最終判定

現状のリポジトリは、卒業研究案を「完全には満たしていない」が、「卒論として十分に成立しうる中核実装」には到達している。

より正確には、

  • 題目の完成形そのもの ではない
  • しかし 「LLM を用いた安全な応急復旧実験基盤と、そのマルチエージェント化・実機PoCへの拡張」 としてはかなり良い状態

である。

卒論本文では、現状システムを

提案システムの第一段階実装
もしくは
DCインフラ応急修復システムに向けた安全な研究基盤・要素実証

と位置付けると、内容と題目の整合が取りやすい。

8. 題目に近づけるための優先課題

優先度順に挙げると、次の 5 点が重要である。

  1. マルチエージェントの本格化
    planner/reviewer/judge の直列ループから進めて、観測担当、根因推定担当、修復候補担当、検証担当などの専門化を行う。

  2. DCインフラらしい対象の追加
    単一 Docker Compose だけでなく、複数ノード、ネットワーク設定、冗長化、フェイルオーバ、ストレージ、監視系を含む構成へ拡張する。

  3. production PoC の修復アクション拡張
    restart だけでなく、低リスクな設定切替、runbook 実行、代替経路切替、切り戻しなどを段階的に導入する。

  4. バックアップ/承認フローの実装
    snapshot provider、human approval、段階的 rollback を本実装し、研究上の安全性主張を強める。

  5. hard scenario の multi-agent 実証
    i2/m/n/o/r/u で single-agent を上回ることを定量的に示す。ここが卒論の最も説得力のある比較実験になる。

9. 結論

このリポジトリは、現状では 「安全な LLM 応急復旧研究基盤」 としては強い。一方で、「LLMマルチエージェント協調を用いたDCインフラ応急修復システム」 という題目に対しては、まだ 要素実証から統合システムへ向かう途中段階 にある。

したがって最も妥当な評価は、

  • 研究基盤としては高評価
  • 題目の完成形としては中程度
  • 卒論としては十分戦えるが、主張の置き方を少し丁寧にするべき

である。
