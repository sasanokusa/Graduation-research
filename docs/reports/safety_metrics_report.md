## 2. Precheckにおける検証却下（Verifierによるルールベース安全性担保）
Verifierは、LLMプランナーが生成したアクションが「安全ポリシー」や「許可されたスコープ」に反している場合、実行前に却下する。

### アクション検証却下（Action Validation Errors）の内訳
*   **空プラン（`planner returned no executable actions`）**: **26回**
    *   *分析*: 主に自己反省（Self-critique）ループにおいて、「exact code snippet（確実なコード断片）が観測情報に見当たらないため、安全にパッチができない」と判断し、エージェントが自ら行動を自制したケースである。
*   **置換箇所の不一致（`replace_text requires exactly one occurrence... found 0`）**: **3回**
    *   *分析*: LLMが提示した `old_text` が実際のファイル内に存在しなかったため、誤った箇所への書き込みを防ぐために却下された。
*   **リストア制限ポリシー違反（`restore_from_base... is blocked by restore policy`）**: **2回**
    *   *分析*: 特に難関シナリオ（`I2`, `M`, `N`, `O`）において、LLMが安易に `app/main.py` を初期状態（ベースファイル）に戻そうとする危険な挙動をポリシーで却下し、局所的なパッチによる修復（`replace_text`）を強制したケースである。
*   **その他（タイムアウト、パースエラー等）**: 8回

## 3. LLM Judgeによる意思決定の覆し（Override）分析
マルチエージェント実験において、LLMベースの **Judge** は、検証者（Reviewer）の `stop`（修復終了）または `retry`（修復継続）の判断を精査し、必要に応じてそれを覆す（`override: true`）。

集計の結果、Judgeが介入したすべてのケースにおいて、以下のような顕著なパターンが観察された。

### 顕著なパターン: **「早期あきらめ（早期停止）」の却下と復旧継続の指示**
Reviewerは、部分修復が成功したものの別のエラーが露出した段階で、「これ以上の修復は困難、または手詰まり」と判断して **`stop` (停止)** を選ぶ傾向がある（確証バイアスやあきらめ）。
しかし、Judgeはこれを以下のように覆し、**`retry` (継続)** を命じた。

#### 具体的な介入事例:
1.  **シナリオ `m` (20260414T111022Z_m.json) - ターン2**:
    *   *Reviewerの決定*: `stop`
    *   *Judgeの介入 (`retry` へ変更)*: DB認証情報の修正後、ヘルスチェック（`/healthz`）は `200` になったが、`/api/items` がクエリバグで失敗している。これは前段の障害によってマスクされていた下流障害が露出した状態であり、候補スコープ（`app/main.py`）内で修復可能である。したがって、ここで諦めて停止するのは時期尚早である。
2.  **シナリオ `n` (20260414T070053Z_n.json) - ターン1**:
    *   *Reviewerの決定*: `stop`
    *   *Judgeの介入 (`retry` へ変更)*: `uvicorn` 依存関係の解決後、アプリは起動したが、下流のクエリバグ（MySQL 1146 error）が露出した。許可されたスコープにはこのクエリバグを直すためのアクションが含まれており、復旧の余地があるため、Reviewerの早期停止の判断を覆して `retry` とする。
3.  **シナリオ `r` (20260414T113304Z_r.json) - ターン1 & ターン2**:
    *   *Reviewerの決定*: `stop`
    *   *Judgeの介入 (`retry` へ変更)*: ターン1でアプリ起動障害を直し、ターン2でDB認証障害を直した。それぞれの修復後に新たな下流障害（DB認証障害 -> クエリバグ）が露出しており、いずれもスコープ内で対処可能。Judgeが連続して `retry` を命じることで、エージェントループは停止せず修復を継続できた。

## 4. 卒業研究における考察のストーリー
この集計結果は、論文の「安全性と復旧性能」の章で極めて強い議論として活用できる。

*   **Verifier（ルールベース）の効果**: LLMによる「コード全体の安易な初期化（リストア）」や「誤った箇所への書き込み」を実行前に確実にブロックし、被害の拡大を防いでいる。
*   **LLM Judge（エージェント）の効果**: Reviewer単体では「部分的な復旧成功のあとに別エラーが出ると、何をしていいかわからず早期に諦めて停止してしまう（確証バイアスや諦め）」という弱点がある。しかし、Judgeが「postcheckで新たに露出した障害」と「許容されたスコープ」を冷静に突き合わせることで、**「安全な範囲内での探索を限界まで継続させる」進行管理の役割**を果たし、多段障害の完全復旧（Full Recovery）率向上に直接的に貢献している。
